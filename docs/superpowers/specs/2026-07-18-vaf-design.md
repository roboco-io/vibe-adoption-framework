- 작성일: 2026-07-18
- 상태: 사용자 승인 완료 (심층인터뷰 + 섹션별 승인)
- 참고 자료: `ideation.md`, `research/aws-caf-overview.md` (AWS CAF 3.0 조사)

## 2. CAF 차용 방식

AWS CAF 3.0의 3축 구조(Perspective+Capability / 여정 단계 / Transformation Domain)와
톤앤매너(prescriptive 권고 + 컨설팅적 유연함 + 아웃컴 지향)를 차용하되,
구성요소는 전면 재설계한다(접근안 B). CAF 용어를 그대로 쓰지 않고 VAF 고유 명명을 사용한다.

CAF-AI의 value chain 개념도 차용한다:
**에이전트 역량 → 도입(capability) → Domain 전환 → 비즈니스 아웃컴**.

## 3. 관통 원리 (Foundational Principles)

문서 서두에 명문화하고 모든 perspective의 capability 서술에 반영한다.

1. **에이전트 = 인력 레버리지**: 고수준 인력을 낮은 비용에 대규모로 고용하는 효과.
   바이브 코딩은 기존 소프트웨어 개발의 베스트 프랙티스·안티패턴을 그대로 계승한다(대전제).
2. **오너십 원칙**: 코드는 에이전트가 쓰고 오너십은 사람이 가진다.
   누가 물어도 언제든 답할 수 있어야 한다.
3. **인지부채 관리**: 기술부채와 마찬가지로 제거 대상이 아닌 트레이드오프 관리 대상.
   모든 perspective를 관통하는 설계 원리.

## 4. Perspective 6개

앞 3개는 사람·조직의 관점, 뒤 3개는 에이전트를 위한 기술 관점.

| # | Perspective | 핵심 질문 | 주요 capability (초안) | 이해관계자 |
|---|---|---|---|---|
| 1 | Strategy (전략) | 왜, 무엇을 얻는가 | 도입 전략, 가치·ROI 측정, AsIs/ToBe SDLC 갭 분석, 우선순위 포트폴리오 | 경영진 |
| 2 | People & Ownership (사람과 오너십) | 사람의 역할은 무엇인가 | 오너십 체계, 인지부채 관리, 역할 전환(작성자→오너), 역량 개발, 문화 전환 | 경영진·팀 리드 |
| 3 | Governance & Guardrails (거버넌스와 가드레일) | 무엇을 강제하는가 | 취향/비취향 구분, 정책의 저장소 강제, 보안·컴플라이언스, 비용 관리 | 거버넌스·보안 책임자 |
| 4 | Agent Environment (에이전트 환경) | 에이전트가 일할 수 있는가 | 접근 통로(repo·이슈트래커·인프라), 도구 연결(MCP 등), 실행 환경, 권한 관리 | 플랫폼·인프라 |
| 5 | Knowledge (지식기반) | 에이전트가 아는가 | LLM Wiki 등 지식기반, 리포지토리 분석·문서화, 현안 문서화, 컨텍스트 상시 활용 체계 | 아키텍트·테크리드 |
| 6 | Harness Engineering (하네스 엔지니어링) | 품질을 누가 지키는가 | 규칙 파일(AGENTS.md 등), 스킬·훅, 자동 리뷰, 검증 파이프라인, 가드레일 자동화 | 테크리드·개발팀 |

capability 개수는 perspective당 4~6개, 본문 집필 시 확정한다.

## 5. 도입 여정 4단계

반복 사이클로 서술한다. VAF의 차별점: **진단보다 환경 준비가 먼저다** —
이후 모든 단계를 에이전트와 함께 수행하기 위함.

1. **준비 (Prepare)** — 에이전트 환경 구축. repo·이슈트래커·파일저장소·인프라 접근 통로와 도구 연결.
2. **진단 (Assess)** — 에이전트와의 심층인터뷰로 비즈니스·IT 현안 문서화, 기존 리포지토리 분석·문서화,
   LLM Wiki 등 지식기반 구축, AsIs SDLC 파악과 ToBe 정의, 갭 정리.
3. **파일럿 (Pilot)** — 우선 프로젝트에 규칙·스킬·훅(하네스) 구성, 취향/비취향 정책의 저장소 강제,
   파일럿 팀의 개발 전환과 학습.
4. **전환 (Transform)** — 전사 확산. "사람이 코드를 손으로 쓰지 않는" 상태 정착,
   오너십·인지부채 관리 체계 운영, 성과 측정과 다음 사이클 진입.

## 6. Transformation Domain 4개

| Domain | AsIs → ToBe |
|---|---|
| 코드 생산 (Code) | 사람이 작성 → 에이전트가 작성, 사람은 의도 지시와 승인 |
| 품질 보증 (Quality) | 사람 리뷰·수동 QA → 하네스가 리뷰·가드레일 자동 수행 |
| 지식 (Knowledge) | 암묵지·구두 전승 → 에이전트가 상시 활용하는 문서화된 지식기반 |
| 조직과 역할 (Organization) | 작성자 중심 팀 → 오너·오케스트레이터 중심 팀 |

주: Knowledge는 perspective와 domain 양쪽에 등장한다. perspective는 "에이전트에게 지식을
공급하는 조직 역량", domain은 "지식 관리 방식 자체의 전환 대상"으로 축이 다르며,
본문에서 이 구분을 명시한다.

## 7. VAF 본문 목차

1. **개요 (Executive Summary)** — VAF란 무엇인가, 왜 지금인가, 기대 아웃컴. 경영진이 이것만 읽어도 되는 수준
2. **대전제와 관통 원리** — 관통 원리 3개(에이전트=인력 레버리지, 오너십 원칙, 인지부채 관리)와
   이름 붙은 핵심 개념(하네스 엔지니어링, 취향/비취향 구분) 정의
3. **프레임워크 구조와 Value Chain** — 3축 소개와 관계 다이어그램(mermaid)
4. **Transformation Domain** — 4개 영역의 AsIs→ToBe, domain별 뒷받침 perspective 매핑
5. **Perspective와 Capability** — 본체. perspective별 정의·이해관계자·capability 상세.
   각 capability는 CAF처럼 prescriptive한 권고로 서술
6. **도입 여정** — 단계별 목표·핵심 활동·산출물·완료 기준
7. **부록: 실행 키트** — ① capability 성숙도 어세스먼트 체크리스트 ② 심층인터뷰 질문지
   ③ 진단·파일럿 워크숍 가이드 ④ 용어집

## 9. 서술 톤 가이드 (CAF 조사 결과 반영)

- **Prescriptive 권고**: "~를 수립하라", "~를 고려하라" 형태의 행동 지시.
  단순 개념 설명에 그치지 않는다.
- **컨설팅적 유연함**: "고려하라", "평가하라" 등 조직 맥락을 존중하는 조동사 사용.
  강제가 아닌 가이드로 포지셔닝.
- **아웃컴 지향**: 각 권고를 기대 효과(리드타임 단축, 품질 향상, 비용 구조 변화)와 연결.
- **일반성 유지**: 특정 산업·규모에 종속되지 않는 서술. 사례는 익명·일반화.

## 10. 작업 규칙 (ideation.md 계승)

- VAF를 완성해 나가는 모든 단계에서 심층인터뷰로 컨텍스트를 보충·확인한다.
- 본문 집필 시 perspective별 capability 정의 등 세부 결정도 인터뷰를 거친다.
